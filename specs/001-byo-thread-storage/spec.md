# 功能規格：BYO Thread Storage — 以 Azure Cosmos DB 作為 Foundry Agent 的對話執行緒儲存體

**功能分支**：`001-byo-thread-storage`  
**建立日期**：2026-03-22  
**狀態**：已釐清  
**輸入**：使用者描述："我在 Microsoft Foundry 有自己的 agents 了，想要遵循 BYO Thread Storage in Azure AI Foundry using Python 文章實作 thread storage，將一個指定的 Azure Cosmos DB 作為我 Foundry agent 的 thread 儲存體。"

## 使用者情境與測試 *（必填）*

### 使用者故事 1 - 建立並持久化新對話執行緒（優先順序：P1）

身為一名已擁有 Foundry Agent 的開發者，我希望當使用者與 Agent 開啟一段新對話時，系統會自動在我指定的 Azure Cosmos DB 中建立一筆新的執行緒紀錄，並附帶使用者識別資訊與初始中繼資料，使得對話上下文從第一條訊息起就被持久化到我自管的儲存體中。

**優先順序原因**：這是整個功能的基礎——如果無法建立並持久化執行緒，所有後續的訊息追加、歷史查詢與刪除操作都無法運作。

**獨立測試**：可透過觸發一次新對話來完整測試，驗證 Cosmos DB 中確實產生了一筆含正確結構的執行緒文件，並交付「對話上下文從第一步即被持久化」的價值。

**驗收情境**：

1. **假設** 系統已設定好 Cosmos DB 連線且 Agent 已部署，**當** 使用者向 Agent 發起一段全新對話，**則** 系統在 Cosmos DB 的指定容器中建立一筆新的執行緒文件，其中包含唯一識別碼、使用者識別碼、建立時間戳記，以及空的訊息列表。
2. **假設** 一筆新的執行緒文件剛被建立，**當** 開發者至 Cosmos DB 中查詢該筆文件，**則** 文件結構符合預定義的執行緒資料格式，且所有必要欄位皆完整。

---

### 使用者故事 2 - 追加訊息並保留完整對話歷史（優先順序：P1）

身為一名開發者，我希望每次使用者發送訊息或 Agent 回覆時，該訊息都會被即時追加到 Cosmos DB 中對應的執行緒文件內，使得 Agent 在後續回覆時能夠取回完整對話歷史作為上下文，維持多輪對話的連貫性。

**優先順序原因**：多輪對話的記憶能力是 BYO Thread Storage 的核心價值——讓 Agent 不再是每次回應都從零開始，而是基於完整上下文作出更準確的回覆。

**獨立測試**：可透過在一個已建立的執行緒中連續發送多條訊息來測試，驗證每條訊息皆被追加至 Cosmos DB、Agent 的回覆引用了先前的對話內容，並交付「Agent 具備跨輪對話記憶」的價值。

**驗收情境**：

1. **假設** 一個已存在的執行緒文件在 Cosmos DB 中，**當** 使用者發送一條新訊息，**則** 系統將使用者訊息（包含角色、內容與時間戳記）追加到該執行緒的訊息列表中，並更新執行緒的最後修改時間。
2. **假設** 一條使用者訊息剛被追加到執行緒，**當** 系統取回完整對話歷史並傳送給 Agent 模型，**則** Agent 的回覆也被追加至同一執行緒，且開發者可確認 Cosmos DB 中的訊息列表包含完整的多輪對話紀錄。
3. **假設** 一個執行緒中已有 10 輪以上的對話，**當** 使用者引用第 3 輪的內容再次提問，**則** Agent 能基於完整歷史正確回應，證明上下文完整保留。

---

### 使用者故事 3 - 查詢與擷取指定執行緒（優先順序：P2）

身為一名開發者，我希望能以執行緒識別碼查詢 Cosmos DB 中的特定執行緒及其完整訊息歷史，以便在除錯、稽核、或使用者重新返回對話時，能快速重現先前的對話上下文。

**優先順序原因**：查詢能力是運維與除錯的基礎，也是讓使用者得以回到之前對話的關鍵，但對核心的對話流程（建立與追加）而言屬於支援性功能。

**獨立測試**：可透過建立一個含有已知訊息的執行緒，再以其識別碼查詢來測試，驗證回傳的資料完整且結構正確，並交付「可按需取回任何歷史對話」的價值。

**驗收情境**：

1. **假設** Cosmos DB 中存在一個含有多條訊息的執行緒，**當** 開發者以執行緒識別碼查詢，**則** 系統回傳該執行緒的完整資料，包含所有訊息、中繼資料與時間戳記。
2. **假設** 開發者查詢一個不存在的執行緒識別碼，**當** 系統嘗試擷取該執行緒，**則** 系統回傳清楚的「查無此執行緒」提示，而非拋出未處理的異常。

---

### 使用者故事 4 - 刪除指定執行緒（優先順序：P3）

身為一名開發者，我希望能根據執行緒識別碼從 Cosmos DB 中刪除一個執行緒及其所有訊息，以滿足資料保留規範或使用者主動要求刪除對話紀錄的需求。

**優先順序原因**：資料刪除是合規性（如 GDPR 的被遺忘權）的必要功能，但在初期 MVP 階段的優先度低於建立、追加與查詢。

**獨立測試**：可透過先建立一個執行緒、再對其執行刪除操作來測試，驗證該執行緒在 Cosmos DB 中確實被移除，並交付「依規範移除對話資料」的價值。

**驗收情境**：

1. **假設** Cosmos DB 中存在一個指定的執行緒，**當** 開發者對該執行緒執行刪除操作，**則** 該執行緒及其所有訊息從 Cosmos DB 中永久移除。
2. **假設** 一個執行緒已被刪除，**當** 開發者再次以相同識別碼查詢，**則** 系統回傳「查無此執行緒」的結果。

---

### 邊界情境

- 當 Cosmos DB 服務暫時無法連線時，系統應如何處理？應回傳明確的錯誤訊息，而非靜默失敗或資料遺失。
- 當兩個並行請求同時對同一個執行緒追加訊息時，系統使用 ETag 樂觀並行控制，衝突時自動重試（最多 3 次）；訊息最終順序以實際寫入 Cosmos DB 的時間為準。
- 當執行緒中的訊息數量極大（例如超過 1,000 條）時，系統的查詢與追加效能是否仍可接受？已知風險：資料模型分析顯示單一文件可容納約 4,000 條訊息（Cosmos DB 2 MB 上限），對絕大多數對話場景已足夠；若未來需要處理超長對話，可在 `append_message` 中加入文件大小預檢。
- 當提供的 Cosmos DB 端點或認證無效時，系統應在初始化階段就報告清楚的連線失敗訊息。
- 當嘗試對一個不存在的執行緒追加訊息時，系統應回傳明確的錯誤提示。
- 當使用者嘗試存取不屬於自己的執行緒時，系統統一回傳「查無此執行緒」錯誤。由於分割鍵為 `user_id`，point read 以 `(thread_id, user_id)` 查詢時，不匹配的 `user_id` 自然得到 404，天然隱藏了執行緒的存在性，此行為比顯式回傳「存取被拒絕」更安全。

## 需求 *（必填）*

### 功能性需求

- **FR-001**：系統必須能以使用者識別碼與選用的中繼資料在 Cosmos DB 中建立一個新的對話執行緒，並回傳唯一的執行緒識別碼。
- **FR-002**：系統必須能以角色（系統、使用者、助理）、內容與時間戳記將訊息追加到指定的執行緒中，並同步更新執行緒的最後修改時間。追加操作使用 ETag 樂觀並行控制；當發生 ETag 衝突時（並行寫入同一執行緒），系統應自動重試，最多重試 3 次；若仍失敗則拋出錯誤。當提供的角色不是 "system"、"user"、"assistant" 之一時，系統應立即拒絕並拋出錯誤。
- **FR-003**：系統必須能以執行緒識別碼擷取完整的執行緒資料，包含全部訊息歷史與中繼資料。
- **FR-004**：系統必須能以執行緒識別碼僅擷取該執行緒的訊息列表（不含完整執行緒中繼資料）。
- **FR-005**：系統必須能以執行緒識別碼從 Cosmos DB 中刪除指定的執行緒及其所有訊息。
- **FR-006**：系統必須將儲存層（Cosmos DB 操作）與 Foundry Agent 對話管理整合，使每次對話互動都自動經由自管儲存體持久化。
- **FR-007**：系統必須在與 Agent 模型互動前，從儲存體取回完整對話歷史。儲存層始終回傳全量訊息，上下文截斷或摘要處理係 Agent 呼叫端或 Foundry 框架的責任，不屬於儲存層的範疇。
- **FR-008**：系統必須使用 Azure 身分驗證機制連線至 Cosmos DB，不得在程式碼中硬編碼任何認證資訊。
- **FR-009**：系統必須在 Cosmos DB 操作失敗時（如連線逾時、文件不存在）提供清楚的錯誤提示，而非拋出未處理的異常。
- **FR-010**：系統必須能以使用者識別碼列出該使用者的所有對話執行緒摘要（包含執行緒識別碼、建立時間、最後更新時間），回傳型別化的 `ThreadSummary` 物件列表（遵循憲章原則 II：型別安全 Python），以支援使用者回溯歷史對話清單。
- **FR-011**：系統必須在初始化時自動偵測 Cosmos DB 中指定的資料庫與容器是否存在，若不存在則自動建立（包含正確的分割鍵設定），開發者僅需提供 Cosmos DB 端點、資料庫名稱與容器名稱即可開始使用。
- **FR-012**：系統必須透過環境變數接收所有必要的設定值（Cosmos DB 端點、資料庫名稱、容器名稱、Foundry 專案連線資訊），以支援 `.env` 檔案與各類部署平台的設定機制。
- **FR-013**：儲存層必須強制使用者資料隔離。所有執行緒的讀取、追加訊息、刪除操作皆必須同時提供 `user_id`（分割鍵），Cosmos DB 的 point read 以 `(thread_id, user_id)` 查詢天然實現隔離；若 `user_id` 不匹配，統一回傳「查無此執行緒」錯誤，以隱藏執行緒存在性。

### 關鍵實體

- **Thread（對話執行緒）**：代表一段完整的對話上下文。關鍵屬性包含唯一識別碼（`id`）、所屬使用者識別碼（`user_id`，同時作為分割鍵）、訊息列表、建立時間、最後更新時間、以及選用的中繼資料。支援以 `user_id` 查詢列出該使用者的所有執行緒。
- **ThreadSummary（執行緒摘要）**：代表 `list_threads` 回傳的輕量執行緒摘要。關鍵屬性包含執行緒識別碼（`id`）、使用者識別碼（`user_id`）、建立時間、最後更新時間、以及選用的中繼資料。不含訊息列表，以減少傳輸量。
- **Message（訊息）**：代表執行緒中的一條對話訊息。關鍵屬性包含角色（系統/使用者/助理）、內容、以及時間戳記。屬於一個 Thread。

## 成功標準 *（必填）*

### 可衡量的成果

- **SC-001**：開發者可在不超過 5 個操作步驟（安裝依賴、設定環境變數、執行初始化、建立執行緒、驗證結果）內完成儲存體連線設定並成功建立第一個執行緒，且整體耗時不超過 5 分鐘。
- **SC-002**：使用者與 Agent 進行多輪對話時，Agent 的回覆能正確引用先前輪次的上下文，證明對話歷史被完整保留。
- **SC-003**：所有執行緒的 CRUD（建立、讀取、更新、刪除）操作在同區域部署環境下，使用者感知的回應時間不超過 3 秒。
- **SC-004**：系統在 Cosmos DB 連線異常時，100% 回傳可理解的錯誤訊息，而非未處理的異常或靜默失敗。
- **SC-005**：所有認證資訊皆透過 Azure 身分驗證機制處理，程式碼中不含任何硬編碼的金鑰或連線字串。

## 釐清

### 會話 2026-03-22

- Q: 使用者是否需要查看自己的歷史對話清單？（影響分割鍵設計） → A: 需要。提供「以使用者 ID 列出該使用者所有執行緒」的功能，分割鍵設計需配合。
- Q: Cosmos DB 容器的初始化策略？程式碼自動建立還是開發者手動建立？ → A: 自動建立。程式碼在初始化時自動偵測並建立資料庫與容器（若不存在），開發者只需提供端點和名稱。
- Q: 對話歷史過長時的上下文管理策略？ → A: 全量送入，由呼叫端管理。儲存層負責完整保存與回傳所有歷史，上下文截斷/摘要由 Agent 呼叫端或 Foundry 框架處理。
- Q: 設定值的傳遞方式？ → A: 環境變數。透過環境變數傳遞所有設定值，開發者可搭配 `.env` 檔案或部署平台的設定機制。
- Q: 不同使用者之間的資料隔離層級？ → A: 儲存層強制隔離。所有讀寫操作同時需要 `user_id`，儲存層驗證執行緒確實屬於該使用者才執行操作。

### 分析修正 2026-03-22（/speckit.analyze 後）

- F1+F3: EC-006 與 FR-013 原要求回傳「存取被拒絕」，但 `user_id` 分割鍵設計下 point read 不匹配時天然回傳 404，無法區分「不存在」與「非所屬」。統一為 `ThreadNotFoundError`，此行為實際上更安全（隱藏執行緒存在性）。
- F2: `append_message` 契約中的 `AccessDeniedError` 在分割鍵設計下永遠不會觸發，已從 Raises 中移除。`AccessDeniedError` 類別保留但標註為未來 RBAC 預留。
- F4: EC-002 並行追加訊息的一致性問題未定義。已補充至 FR-002：ETag 樂觀並行控制，衝突時最多重試 3 次。
- F5: SC-003（CRUD < 3 秒）缺乏專屬驗證任務。已在 tasks.md T022 中新增 Python `time` 模組計時驗證步驟。
- F6: SC-003「一般網路條件下」定義模糊。已修正為「同區域部署環境下」。
- F7: EC-003（>1,000 條訊息效能）已標註為已知風險，引用 data-model.md 的 2 MB / ~4,000 條分析。
- F8: 規格狀態從「草稿」更新為「已釐清」。

### 會話 2026-03-22（憲章合規釐清）

- F3（中）：SC-001「5 分鐘」缺乏可自動化驗證的指標。已修正為「不超過 5 個操作步驟（安裝依賴、設定環境變數、執行初始化、建立執行緒、驗證結果）且整體耗時不超過 5 分鐘」，T022 新增步驟計數驗證。

## 假設

- 開發者已擁有一個可運作的 Azure AI Foundry 專案，且其中已有至少一個已部署的 Agent。
- 開發者已擁有一個 Azure Cosmos DB 帳號，並有權限建立資料庫與容器（程式碼會自動建立，無需手動預先建立）。
- 開發者的執行環境已設定 Azure 身分驗證（例如 Azure CLI 登入或 Managed Identity），無需額外的認證設定教學。
- 本功能聚焦於單一區域部署；跨區域災難復原（DR）屬於進階功能，不在本次範圍內。
- 對話歷史的上下文截斷或摘要不屬於儲存層的責任。儲存層始終保存與回傳完整歷史，上下文管理由 Agent 呼叫端處理。
- 對話執行緒的分割鍵（Partition Key）採用 `user_id` 作為分割鍵，以同時支援「以使用者 ID 列出所有執行緒」與「以執行緒 ID 讀寫單一執行緒」的存取模式。
